Framing the problem

Open-weight is not open source — and not trust by default

Open-weight gives you more control over execution. It does not, on its own, give you trust in the model or the AI system built around it.

What open-weight DOES give

  • ✓ Weights are accessible and can be deployed on controlled infrastructure
  • ✓ Less dependency on a single external API provider
  • ✓ More sovereignty over where and how inference runs

What it does NOT give

  • ✗ Training data, training code, eval methods or reproducibility
  • ✗ Evidence the model is unbiased, safe, lawful or robust
  • ✗ Any guarantee it is free from hidden behaviors
The real question

Not “Can we host this model?” but “Can we trust the results of this AI system for this use case?”

The hard problem

The real risk is results integrity — not just the model

Integrity must hold across the whole chain, not the model alone.

!

A correct model + wrong, outdated or poisoned context = a wrong result.

!

A safe model + poorly controlled prompt, RAG or tools = unsafe behavior.

!

A well-hosted model can still be biased, misleading or low-quality.

⚙

For agents: was the right tool called, with the right permissions, at the right time?

Goal: approve an AI system for a use case, data scope & autonomy level — not certify a model in isolation.

Where integrity can fail
🤖Model integrity
📦Packaging integrity
📝Prompt integrity
📚Context & RAG integrity
🔧Tool & agent integrity
📊Output integrity
👀Monitoring integrity

Grade the trust

Trust must be graded, not assumed

Trust should be proportional to use-case criticality and the level of evidence available — from a vendor claim to continuous assurance.

L0Vendor claim onlyNo production use
L1Basic documentationLow-risk experimentation
L2Supply chain evidenceInternal productivity
L3Independent testingSensitive internal use
L4Contractual assuranceRegulated / business-critical
L5Continuous assuranceHigh-impact production

Combine both: an approved-model catalogue × use-case-specific approval.

Architecture

Defense-in-depth for open-weight AI

Layered controls across the full lifecycle and runtime — no single control carries the trust.

📥
Intake

Approved sources & trusted repos; licence & intended-use review.

🔎
Artifact security

Malware/serialization scans; hashes, signatures, AI-BOM.

🔒
Runtime

Sandboxed load; hardened containers; network & secret isolation.

📁
Data

Least privilege for RAG, search & document retrieval.

🧰
Prompt & tools

System prompts & tool permissions governed as sensitive config.

🛡
Output

Guardrails, DLP, policy validation, human review.

📈
Monitoring

Drift, anomalies, unexpected tool use, quality decay.

🔬

For high-risk use cases, add internal model observability to detect deeper behavioral deviations — not just bad inputs and outputs.

Enterprise methodology

From intake to production approval

A repeatable homologation process for open-weight AI systems — the same path every time.

01
IntakePurpose, users, data types, decisions, autonomy level.
02
TieringClassify by impact, data sensitivity, regulation, automation.
03
Due diligenceProvenance, licence, intended use, known limits.
04
Supply chainHashes, signatures, AI-BOM, tokenizer, deps, scripts.
05
Security & safety testsBias, toxicity, hallucination, jailbreak, prompt injection.
06
Architecture reviewHosting, identity, flows, RAG, logging, DLP, secrets.
07
Production gateBusiness, cyber, legal, data & risk sign-off.
08
Continuous assuranceMonitor; revalidate on any change.

📋 Evidence pack

Document decisions, tests, residual risks and compensating controls before the gate.

🤝 Cross-functional gate

No single owner approves: business, cyber, legal, data and risk decide together.

🔄 Reassess on change

Model, prompt, RAG source, toolset or use-case change → trigger re-homologation.

Implementation roadmap

Build the capability progressively

0–3 MONTHS · FRAME
  • • Open-weight usage policy
  • • Risk tiers (data, impact, autonomy)
  • • Approval workflow for high-risk
  • • Find shadow deployments
  • • Min. controls: sources, sandbox, scanning, restricted runtime
3–6 MONTHS · INDUSTRIALIZE
  • • Enterprise model catalogue
  • • AI gateway for prompts/tools/RAG/outputs
  • • DLP, logging, monitoring, escalation
  • • AI Security Review Board
6–12 MONTHS · SCALE
  • • Automated evidence packs & reassessment
  • • Integrate with SOC, GRC, model registry
  • • Tools: scanning, runtime, observability
  • • Extend to agentic & tool-calling
  • • Project-by-project → continuous assurance
