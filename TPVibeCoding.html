A

Expliquer le concept

Le vibe coding devient simple quand les participants voient clairement le flux : idée, code, historique, publication.

Brique 1

Agent IA de code

Codex ou Claude transforme une intention en fichiers HTML, CSS et JavaScript. Le rôle du participant est de formuler, tester, corriger et orienter.

→
Brique 2

Repository GitHub

Le code est stocké, historisé et partageable. Une repository est simplement le dossier central du projet, avec l'historique de ses changements.

→
Brique 3

Publication web

GitHub Pages publie automatiquement le contenu de la branche principale. Le prototype devient une URL accessible depuis un navigateur.

Message animateur : le but n'est pas de tout comprendre à Git dès le premier jour. Le but est de comprendre le circuit et d'oser itérer.

B

Configurer les environnements

Préparer les comptes, créer la repository, activer GitHub Pages, puis connecter Codex à GitHub.

1

Vérifier les prérequis

Demander aux participants d'avoir un email accessible. Un compte Google/Gmail est pratique pour l'authentification, mais l'essentiel est de pouvoir recevoir les emails GitHub et ChatGPT.

  • Ouvrir github.com.
  • Ouvrir chatgpt.com.
  • Vérifier que chaque participant peut se connecter ou créer un compte.
2

Créer la repository GitHub

Expliquer le vocabulaire : une repository est le projet GitHub. Elle contiendra le code, l'historique et les futures Pull Requests.

  • Dans GitHub, cliquer sur le bouton + en haut à droite, puis New repository.
  • Nommer la repository, par exemple cyber-snake-academy.
  • Choisir Public pour simplifier GitHub Pages pendant l'atelier.
  • Cocher Add a README file pour initialiser le projet.
  • Cliquer sur Create repository.
Capture à préparerGitHub / New repository

Faire une capture du formulaire avec le nom de repository, l'option Public et la case Add a README file cochée.

3

Activer GitHub Pages

GitHub Pages va publier automatiquement le contenu statique de la branche main. Pour ce TP, publier depuis la racine du projet est le chemin le plus simple.

  • Dans la repository, cliquer sur Settings.
  • Dans le menu de gauche, cliquer sur Pages.
  • Dans Build and deployment, choisir Source : Deploy from a branch.
  • Dans Branch, sélectionner main puis /root.
  • Cliquer sur Save et noter l'URL qui sera affichée après déploiement.
Capture à préparerSettings / Pages

Capturer l'écran où l'on voit Source = Deploy from a branch, Branch = main et dossier = /root.

4

Lancer Codex et connecter GitHub

Codex web permet de travailler avec les repositories GitHub et de créer des Pull Requests depuis son travail. L'accès peut dépendre du plan ChatGPT ou des règles de l'espace de travail.

  • Aller sur chatgpt.com/codex.
  • Se connecter au compte ChatGPT.
  • Cliquer sur Connect GitHub ou Connect account si demandé.
  • Autoriser l'accès GitHub, idéalement uniquement à la repository du TP.
  • Vérifier que la repository apparaît dans la liste des projets disponibles.
Capture à préparerChatGPT / Codex / GitHub

Capturer l'entrée Codex dans ChatGPT, puis l'écran d'autorisation GitHub si l'organisation autorise la capture.

C

Commencer à développer

Créer le premier prompt, laisser Codex produire le jeu, relire les changements, ouvrir une Pull Request, la fusionner et vérifier la publication.

Astuce animateur

Dire ce que l'on veut vraiment

Le prompt doit décrire le résultat attendu, le public, les règles du jeu, les critères de réussite et les contraintes techniques. Ce n'est pas une commande magique : c'est un brief de réalisation.

Vocabulaire

Pull Request

Une Pull Request est une proposition de modification. Elle permet de relire les fichiers changés avant de les fusionner dans main, la branche publiée par GitHub Pages.

Je suis en charge de notre Cyber Academy. Pour valider les compétences acquises à la fin des modules, je voudrais une approche gamifiée.

Crée une page web autonome en HTML/CSS/JavaScript qui propose un jeu de serpent. Chaque fois que le serpent mange une brique, le jeu se met en pause et pose une question de cybersécurité avec 3 choix :
- 1 réponse clairement fausse ;
- 1 réponse presque bonne mais incomplète ;
- 1 bonne réponse.

Prépare 30 questions tirées au hasard sur l'IAM, la GRC, le SOC et la sécurité de l'IA. Pendant une partie, pose 10 questions au maximum et affiche le score final. Si le joueur obtient au moins 8 bonnes réponses sur 10, affiche "Compétence validée". Sinon, propose de recommencer.

Contraintes :
- tout doit tenir dans une seule page index.html ;
- le jeu doit fonctionner sur GitHub Pages ;
- l'interface doit être claire, professionnelle et utilisable en atelier ;
- ajoute un court écran d'accueil et un écran de score final.
Ajoute une animation de feu d'artifice à chaque bonne réponse, sans perturber la lisibilité de la question. Ajoute aussi un bouton pour couper les animations si l'utilisateur préfère une interface plus sobre.
Crée un framework HTML interactif pour mesurer la conformité au document CSA "Are You Mythos Ready?". Je veux une grille de critères, un score de maturité, des recommandations automatiques et une synthèse exportable.
1

Créer une tâche Codex

Démarrer un nouveau chat ou une nouvelle tâche Codex, sélectionner la repository du TP, puis coller le prompt principal.

  • Dans Codex, cliquer sur New task ou New chat.
  • Sélectionner la repository cyber-snake-academy.
  • Coller le prompt principal et lancer la tâche.
  • Attendre que Codex propose les fichiers et relire rapidement le résultat.
Capture à préparerCodex / New task

Capturer l'écran avec la sélection de repository et le prompt prêt à être envoyé.

2

Créer la Pull Request

Quand le résultat semble correct, demander à Codex de préparer une Pull Request. Expliquer que c'est une proposition de changement vers le système central, pas une publication directe.

  • Dans Codex, ouvrir le récapitulatif des changements.
  • Cliquer sur Create pull request, Push branch ou l'action équivalente affichée.
  • Donner un titre simple, par exemple Add cybersecurity snake game.
  • Ouvrir le lien GitHub de la Pull Request.
Capture à préparerCodex / Pull Request

Capturer l'écran où Codex affiche les fichiers modifiés et le bouton de création de Pull Request.

3

Valider la Pull Request dans GitHub

Montrer la revue minimale : fichiers modifiés, aperçu du code, puis fusion dans main. C'est cette fusion qui déclenche ensuite la publication GitHub Pages.

  • Dans GitHub, ouvrir l'onglet Pull requests.
  • Cliquer sur la Pull Request créée par Codex.
  • Ouvrir Files changed pour montrer ce qui a été ajouté.
  • Revenir sur Conversation, cliquer sur Merge pull request, puis Confirm merge.
Capture à préparerGitHub / Merge PR

Capturer la Pull Request avec le bouton Merge pull request visible, puis l'état merged.

4

Attendre la mise en ligne

GitHub Pages peut prendre quelques secondes à quelques minutes. Pendant l'attente, montrer où vérifier le statut du déploiement.

  • Dans la repository, ouvrir l'onglet Actions pour voir le workflow Pages.
  • Ou retourner dans Settings, puis Pages, pour vérifier l'URL publiée.
  • Quand le déploiement est vert, ouvrir l'URL GitHub Pages.
  • Tester le jeu avec les participants et noter les idées d'amélioration.
Capture à préparerGitHub / Actions / Pages

Capturer soit l'exécution GitHub Actions réussie, soit Settings > Pages avec le message indiquant que le site est live.

D

Et recommencer

Le vrai déclic vient après la première publication : on peut améliorer, enrichir, corriger, tester, puis republier.

Effet waouh

Ajouter des effets

Feux d'artifice, transitions, sons optionnels, mode sombre, animations de score, feedback visuel.

Contenu

Étendre les questions

Ajouter les SI industriels, la continuité d'activité, le cloud, la privacy, la cryptographie ou les risques fournisseurs.

Nouveau projet

Changer d'objet

Créer un questionnaire de maturité, un simulateur de crise, une matrice de conformité ou un générateur de fiches pédagogiques.

E

Pour aller plus loin

Il n'y a pas de limite claire tant que l'on sait dialoguer avec l'IA, tester le résultat et garder une revue humaine.

Si vous êtes bloqués, demandez à l'IA : "Explique-moi l'erreur", "Propose trois façons de simplifier", "Pose-moi des questions pour clarifier mon besoin".

Gardez un réflexe de revue : ne publiez pas de données sensibles, vérifiez les textes générés, testez le parcours et relisez les changements avant de fusionner.
